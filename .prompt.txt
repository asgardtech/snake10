### 🔴 Critical: Behavioral Regression in `restartGame()`

The restart behavior has changed in a way that degrades UX:

**Before (original code):**
```javascript
const restartGame = () => {
  engineRef.current.reset();
  setScore(0);
  setGameOver(false);
  startGame();  // ← Immediately starts game
};
```

**After (this PR):**
```javascript
const restartGame = () => {
  engineRef.current.reset();
  setScore(0);
  setGameState('idle');  // ← Returns to idle, requires another click
};
```

After a game over, clicking "Restart" now returns to idle state instead of immediately resuming gameplay. Users must click "Start Game" in addition to "Restart", requiring an extra interaction.

**Recommendation:** Call `startGame()` at the end of `restartGame()` to match the original behavior:
```javascript
const restartGame = () => {
  if (!engineRef.current) return;
  engineRef.current.reset();
  setScore(0);
  startGame();  // or setGameState('playing')
};
```

2. Extract the code into the promised files for better modularity and reusability

---

### 2. **Semantic Mismatch Between Game State and Engine State** ⚠️
The Game component defines:
```typescript
type GameState = 'idle' | 'playing' | 'paused' | 'gameOver';
```

But `GameEngine` only has:
```typescript
type GameState = 'idle' | 'playing' | 'gameOver';
```

When `pauseGame()` is called, the Game component sets `gameState = 'paused'`, but `GameEngine.pause()` sets its state to `'idle'`. This means the two state machines become temporarily out of sync:
- Game thinks: `paused`
- Engine thinks: `idle`

While this works functionally (because the game loop checks `gameState === 'playing'`), it's a design smell that could cause confusion and bugs in future refactoring.

**Recommendation:** Either document this design decision with a comment, or update `GameEngine` to support the 'paused' state explicitly for better semantic alignment.

---

### 3. **Missing useGameLoop Return Assignment** ⚠️
The old code was:
```typescript
const { stop } = useGameLoop({ ... });
```

The new code is:
```typescript
useGameLoop({ ... });
```

The `stop()` function is no longer used. **This is actually fine** — it's a code improvement since the loop is now controlled by the canvas parameter, not by a stop callback. But verify that the `useGameLoop` hook doesn't expect the return value to be used.
